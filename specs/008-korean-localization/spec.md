# Feature Specification: 한국어 용어 전환

**Feature Branch**: `008-korean-localization`
**Created**: 2026-02-18
**Status**: Draft
**Input**: User description: "Change english terminology into Korean. And change 'Wishlist' to '맛집'"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 모든 UI 텍스트가 한국어로 표시됨 (Priority: P1)

사용자가 앱의 모든 화면(검색, 맛집 목록, 음식점 상세, 로그인, 마이페이지)을 탐색할 때, 모든 버튼 라벨, 제목, 안내 문구, 상태 메시지가 한국어로 표시되어야 한다. 기존의 "Wishlist"라는 용어는 모든 곳에서 "맛집"으로 대체된다.

**Why this priority**: 앱의 핵심 사용자 경험이며, 전체 기능에 걸친 변경사항이다. 한국어 사용자가 자연스러운 모국어 인터페이스를 경험하는 것이 가장 중요하다.

**Independent Test**: 앱의 각 페이지를 방문하여 영문 텍스트가 남아 있지 않은지 확인하고, "Wishlist" 대신 "맛집"이 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 하단 내비게이션을 볼 때, **When** 탭 라벨을 확인하면, **Then** "맛집", "검색", "MY" 라벨이 한국어로 표시된다
2. **Given** 사용자가 맛집 목록 페이지에 진입할 때, **When** 페이지가 로드되면, **Then** 제목이 "나의 맛집"으로 표시된다
3. **Given** 사용자가 맛집 목록이 비어있을 때, **When** 페이지를 보면, **Then** "아직 저장된 맛집이 없습니다" 등 한국어 안내 문구가 표시된다
4. **Given** 사용자가 검색 결과에서 음식점 카드를 볼 때, **When** 저장 버튼을 확인하면, **Then** "맛집 추가" 또는 "저장됨" 등 한국어 라벨이 표시된다

---

### User Story 2 - 로그인 화면 한국어 표시 (Priority: P1)

사용자가 로그인 페이지를 방문하면 앱 이름, 설명, 버튼 텍스트, 오류 메시지가 모두 한국어로 표시된다.

**Why this priority**: 로그인은 사용자가 처음 접하는 화면이므로 첫인상이 중요하다.

**Independent Test**: 로그인 페이지에서 모든 텍스트가 한국어인지 확인하고, 로그인 실패 시 오류 메시지도 한국어인지 확인한다.

**Acceptance Scenarios**:

1. **Given** 미인증 사용자가 로그인 페이지에 진입할 때, **When** 페이지가 로드되면, **Then** "맛집 리스트", "카카오로 로그인" 등 한국어 텍스트가 표시된다
2. **Given** 로그인 중 오류가 발생할 때, **When** 오류 메시지가 표시되면, **Then** "로그인에 실패했습니다" 등 한국어 메시지가 표시된다

---

### User Story 3 - 음식점 상세 및 마이페이지 한국어 표시 (Priority: P2)

사용자가 음식점 상세 페이지와 마이페이지를 방문하면 모든 라벨, 버튼, 상태 메시지가 한국어로 표시된다.

**Why this priority**: 주요 기능 페이지이나 핵심 플로우(검색, 맛집 목록)보다 방문 빈도가 낮다.

**Independent Test**: 음식점 상세 페이지에서 "뒤로가기", "평점", "맛집에서 삭제", "카카오맵에서 보기" 등이 한국어인지 확인한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 음식점 상세 페이지를 볼 때, **When** 버튼과 라벨을 확인하면, **Then** "← 뒤로", "평점:", "맛집에서 삭제", "카카오맵에서 보기" 등이 한국어로 표시된다
2. **Given** 사용자가 마이페이지를 볼 때, **When** 페이지가 로드되면, **Then** "내 정보", "로그아웃" 등이 한국어로 표시된다

---

### Edge Cases

- 로딩 상태 텍스트("로딩 중...")가 한국어인가?
- 접근성 라벨(aria-label)도 한국어로 변경되는가?
- 브라우저 탭 제목(HTML title, meta description)도 한국어로 변경되는가?
- 에러 토스트 메시지("검색에 실패했습니다. 다시 시도해 주세요.")가 한국어인가?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 모든 사용자 대면 텍스트(버튼 라벨, 제목, 안내 문구, placeholder, 상태 메시지)를 한국어로 표시해야 한다
- **FR-002**: 기존 "Wishlist" 용어는 모든 곳에서 "맛집"으로 대체되어야 한다 (UI 텍스트, 페이지 제목, 브라우저 탭 제목 포함)
- **FR-003**: 로그인 페이지의 앱 이름, 설명, 버튼 텍스트, 오류 메시지가 한국어로 표시되어야 한다
- **FR-004**: 검색 관련 UI(검색창 placeholder, "이 지역 검색" 버튼, 오류 토스트)가 한국어로 표시되어야 한다
- **FR-005**: 접근성 라벨(aria-label, role 속성의 라벨)도 한국어로 변경되어야 한다
- **FR-006**: HTML 메타데이터(title, description)가 한국어로 변경되어야 한다
- **FR-007**: 음식점 카드의 저장 상태 표시("저장됨" / "맛집 추가" / "삭제")가 한국어로 표시되어야 한다

### Key Entities

- **UI 텍스트 문자열**: 앱 전반에 하드코딩된 영문 문자열 약 40개 이상. 내비게이션, 페이지 제목, 버튼, placeholder, 오류 메시지, 접근성 라벨을 포함
- **용어 매핑**: "Wishlist" → "맛집", "Search" → "검색", "My" / "My Info" → "MY" / "내 정보" 등 핵심 용어의 일대일 대응 관계

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 앱의 모든 페이지에서 사용자 대면 영문 텍스트가 0건이어야 한다 (동적 데이터 및 고유명사 제외)
- **SC-002**: "Wishlist"라는 단어가 UI에서 완전히 제거되고 "맛집"으로 대체되어야 한다
- **SC-003**: 모든 접근성 라벨(aria-label)이 한국어로 제공되어 스크린 리더 사용자도 한국어로 안내받을 수 있어야 한다
- **SC-004**: 브라우저 탭 제목과 메타 설명이 한국어로 표시되어야 한다

## Assumptions

- 이번 변경은 다국어(i18n) 프레임워크 도입이 아닌, 기존 영문 하드코딩 텍스트를 한국어 하드코딩 텍스트로 직접 교체하는 작업이다
- 카카오 API에서 반환하는 동적 데이터(음식점 이름, 카테고리명 등)는 이미 한국어이므로 변경 대상이 아니다
- 코드 내부 변수명, 함수명, 주석 등 개발자 대면 텍스트는 영문을 유지한다
- "MY" 탭 라벨은 한국 앱에서 관례적으로 사용되는 표현이므로 그대로 유지한다
